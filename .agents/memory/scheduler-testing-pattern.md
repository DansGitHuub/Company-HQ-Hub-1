---
name: Testing new cron/scheduler modules
description: How to actually exercise a new server/*Scheduler.ts file end-to-end without a broken standalone test script
---

Scheduler files in this codebase (e.g. `notificationScheduler.ts`, `invoiceOverdueScheduler.ts`)
commonly import `log` from `./index` (server/index.ts). server/index.ts has top-level
side effects (creates the Express app and calls `app.listen`). Importing any module
that imports `./index` — even indirectly, even just for the `log` helper — re-runs
the entire server bootstrap.

**Why:** A standalone `tsx some-test-script.ts` that imports a scheduler module will
try to bind port 5000 a second time and crash with `EADDRINUSE`, because the real
workflow is already listening. There is no way to import just "part of" server/index.ts.

**How to apply:** Do not try to unit-test a new scheduler function via a throwaway
tsx script. Instead:
1. Insert throwaway test data via SQL.
2. Restart the actual "Start application" workflow — this resets any in-memory
   "already ran today" guard and lets the scheduler's normal post-boot timer
   (if it has one) or its natural interval fire against real data.
3. Query the resulting DB rows / hit the real API to confirm behavior.
4. Delete the test data (and restart once more) before finishing, since the
   scheduler will act on real production rows (e.g. real admins may receive a
   real in-app notification generated by your test employee/job — clean those
   notification rows up specifically, not just your fake users).
