# CompanyHQ App Cleanup, Connection, and Build Control Plan

## Main Goal

We are not trying to blindly add more features.

The goal is to make CompanyHQ a complete, connected, simple-to-use operating system for a landscape / contracting business.

The app should handle everything except:

* Landscape/design creation
* Collecting payments
* Making payments
* Full business accounting

QuickBooks and other outside tools may still handle accounting, payments, payroll processing, and other external functions.

CompanyHQ must become:

* Properly connected
* Easy to use
* Cleanly organized
* Safe from duplicate systems
* Reliable for field crews
* Reliable for office/admin
* Reliable for customer portal use
* Capable of growing into a website, Android app, or iOS app later

Before adding any new feature, Replit must always double-check that the feature:

1. Does not duplicate an existing feature
2. Does not hide or bury a wanted feature
3. Does not add unnecessary menu clutter
4. Connects to the correct database tables
5. Connects to the correct workflow
6. Has proper role permissions
7. Has a clear owner/user
8. Has a visible place in the UI
9. Has a clear reason to exist
10. Does not miss the actual business workflow

No feature should be added just because it sounds useful.

# Phase 0 — Stop and Clean the Foundation First

Do not build more major features until these are complete.

## 1. Remove `storedPassword`

### Decision

Remove it immediately.

### Why

Plaintext password storage is unacceptable. Admin convenience is not worth the security risk.

### Replit should:

* Find every reference to `storedPassword`
* Remove it from the schema
* Remove it from admin UI
* Remove it from create/update user logic
* Create a migration to drop the column
* Replace the workflow with password reset / invite flow

### Do not:

* Keep it hidden
* Rename it
* Encrypt and keep it
* Let admins view staff passwords

---

## 2. Add `customer_id` to `jobs`

### Decision

Jobs must directly belong to customers.

### Why

Customer job history, portal visibility, warranty records, closeout records, reports, and future work all depend on this.

### Replit should:

* Add `jobs.customer_id`
* Reference `users.id`
* Backfill existing jobs by matching existing client/contact data
* Keep old text fields temporarily as historical snapshot fields
* Update create/edit job routes
* Update job detail page
* Update customer portal
* Update reports
* Deprecate `customerJobs` after the new FK is confirmed

### Do not:

* Keep relying on customer name text
* Build more customer-facing features without this relationship

---

# Main Menu Rule

The main menu should be treated like prime real estate.

Before adding a tab, Replit must answer:

1. Who uses this?
2. How often?
3. Is it already available somewhere else?
4. Should it be a tab, sub-tab, button, or hidden admin tool?
5. Does it make the app simpler or more complicated?
6. Does it connect to the correct data?
7. Does it have a complete backend?
8. Does it have permissions?
9. Does it show real data?
10. Will this confuse crews, office, or customers?

---

# Connector Rule

All connectors must be properly installed, labeled, tested, and visibly connected.

Replit should audit connectors for:

* QuickBooks
* CompanyCam
* Google Calendar
* Gmail/email
* Google OAuth
* Replit object storage
* Resend/email sending
* future Zapier/webhooks

For each connector:

* What does it connect to?
* What data enters the app?
* What data leaves the app?
* What table stores the external ID?
* What happens if sync fails?
* Is there a health/status page?
* Is there a manual resync button?
* Is it labeled clearly in the UI?
* Who can access it?
* Is it customer-facing or internal only?

---

# Final Replit Instruction

Before making any change, Replit should provide:

1. What problem this solves
2. What existing feature it touches
3. Whether a duplicate system already exists
4. What tables are affected
5. What relationships are created or changed
6. What pages/components are affected
7. What permissions are needed
8. What data migration is required
9. What should be hidden or removed after completion
10. What must be tested before the change is accepted

No new feature should be considered complete unless:

* It has a real database connection
* It appears in the correct UI location
* It has permissions
* It connects to the correct workflow
* It creates no duplicate system
* It has a clear owner/user
* It is simple enough for real staff to use
* It supports the business instead of adding clutter

The app should become powerful enough to run the company, but simple enough that crews, office staff, customers, and managers can use it without feeling like they are inside complicated software.
