SCHEMA (Migration block 13, idempotent — append to server/migrations/companyCamPhase1.ts or a Phase 3 file, whichever matches the existing naming pattern)
- ALTER TABLE customers ADD COLUMN IF NOT EXISTS companycam_project_id TEXT.
- ALTER TABLE customers ADD COLUMN IF NOT EXISTS companycam_create_status TEXT — values: 'created', 'failed', or NULL (NULL = not attempted, e.g. opt-out was off when customer was created, or pre-existing customer).
- ALTER TABLE customers ADD COLUMN IF NOT EXISTS companycam_create_error TEXT — stores last error message when status='failed', NULL otherwise.
- CREATE INDEX IF NOT EXISTS for companycam_create_status WHERE companycam_create_status = 'failed' (partial index — small footprint, fast badge query).
- Drizzle mirror updates in shared/schema.ts. Regenerate Drizzle migration if the project uses generated migrations.

DO NOT ADD a separate companycam_create_status enum type — keep it as TEXT with a comment in the column listing the allowed values. This matches the existing description_source pattern (also TEXT, not enum).

SETTINGS TABLE

If the codebase has a different convention (e.g., a dedicated settings.ts module with a JSON blob in DB), match that. The toggle UI reads/writes through that. Default ON when the row is missing.

SERVER LOGIC
Grep for the customer-create endpoint (likely server/customerRoutes.ts — confirm by grep). After the existing customer INSERT (which must complete and the response must reflect success regardless of CC outcome):

1. Read the global "auto_create_projects" setting. If false → skip the CC call, leave companycam_create_status NULL. Done.
2. If true → fire a non-blocking call to POST https://api.companycam.com/v2/projects with body:
   { name: customer.display_name, address: { street, city, state, postal_code, country: "US" } }
   Use COMPANYCAM_API_TOKEN from process.env, same Authorization header pattern as v1.2.2's delayed-fetch.
3. On success (HTTP 2xx with body containing project id): UPDATE customers SET companycam_project_id = $1, companycam_create_status = 'created', companycam_create_error = NULL WHERE id = $2.
4. On failure (HTTP non-2xx, network error, or token missing): UPDATE customers SET companycam_create_status = 'failed', companycam_create_error = $1 WHERE id = $2. Log the failure with prefix [companycam] customer-create-fail for $customerId: $errorSummary.

The CC API call MUST NOT block the customer-create HTTP response. Pattern: kick off via setTimeout(0, async () => {...}) or Promise.resolve().then(async () => {...}).catch(...), or pass to a fire-and-forget helper. Customer record is committed and returned to the client BEFORE the CC call resolves. Errors in the CC path log + persist to companycam_create_status — they do not surface to the client at create time.

ADMIN RETRY ENDPOINT (in server/customerRoutes.ts or wherever Wave 2's customer logic lives)
POST /api/admin/customers/:id/retry-companycam-project
- Auth: requireAuth + requireRole(...STAFF_ROLES) — same pattern as v11.13's /api/admin/companycam/backfill-descriptions.
- Body: empty.
- Behavior: Look up customer by id. If companycam_project_id already exists, return 200 { skipped: true, project_id: ... } without re-creating. Otherwise re-run the same CC create call as in customer-create flow. Update companycam_create_status / companycam_create_error / companycam_project_id same as above. Return 200 { status: 'created', project_id } or 500 { status: 'failed', error } based on outcome.
- This endpoint CAN return 500 on errors (it's not a webhook receiver — §4.4).

UI: FAILURE BADGE
On the customer detail page, in the same header region where customer name/address show: when companycam_create_status === 'failed', render an inline badge styled muted-red/amber matching existing app conventions, text "CompanyCam: not connected · Retry". The "Retry" word is the click target — it fires the admin retry endpoint, shows a brief spinner, then refetches the customer data to flip the badge to either green-success-disappears or red-still-failed (with the error tooltip showing companycam_create_error if present). Do not show this badge when companycam_create_status is NULL or 'created'.

UI: GLOBAL TOGGLE
In whatever the existing admin/settings page is, add a single labeled toggle: "Auto-create CompanyCam projects when customers are added" with a small subtitle "When on, every new customer record creates a matching CompanyCam project." Default ON. Read/write through the same settings pattern as existing toggles.

DO NOT ADD a "Open in CompanyCam" button on the customer page in this wave. That decision is deferred per §C.3 Q1.

ACCEPTANCE TEST (Dan will drive)
1. With the global toggle ON, create a brand new customer with display_name + valid address. Within ~30s the customer record should have companycam_project_id populated AND a corresponding project visible in CompanyCam mobile.
2. Set the toggle OFF, create another customer. Verify companycam_project_id is NULL and companycam_create_status is NULL. No CC API call fired (check logs).
3. Toggle ON, set COMPANYCAM_API_TOKEN to an invalid value temporarily, create a customer. Customer creation succeeds; companycam_create_status='failed'; failure badge visible on customer header. Restore the token, click Retry — badge disappears, project_id populated.
