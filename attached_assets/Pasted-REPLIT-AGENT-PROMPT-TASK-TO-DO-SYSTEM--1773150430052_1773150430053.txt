REPLIT AGENT PROMPT — TASK & TO-DO SYSTEM
──────────────────────────────────────────────────────────
Rebuild the To-Do List module in CompanyHQ into a complete internal task management system. This is for internal Chapin Landscapes employees only — not customer-facing. Read all sections before writing code. The #1 design requirement is speed: creating a task must take under 10 seconds, and notifications must be impossible to miss.
SECTION 1: DATABASE SCHEMA
Create these tables using Drizzle ORM: tasks — id, task_id (auto TK-XXXX), title (required, max 120 chars), description, type (standard/checklist/delegated/recurring), priority (p1_urgent/p2_high/p3_normal/p4_low), status (assigned/acknowledged/in_progress/on_hold/reassigned/completed/confirmed/cancelled/overdue), created_by_user_id (required), assigned_to_user_id (required), due_date, due_time, category, estimated_minutes, location, requires_confirmation (boolean), completion_note, completion_photo_url, confirmed_by_user_id, cancelled_reason, on_hold_reason, linked_job_id, linked_equipment_id, visibility (private/team), is_recurring (boolean), recurring_config (JSON: frequency, days, interval), parent_task_id (for sub-tasks), created_at, updated_at, acknowledged_at, completed_at, confirmed_at.
task_checklist_items — id, task_id, item_text, is_completed, completed_by, completed_at, sort_order.
task_history — id, task_id, event_type (created/acknowledged/status_changed/reassigned/edited/commented/completed/confirmed/cancelled), changed_by_user_id, old_value, new_value, note, created_at. Never delete from this table.
task_attachments — id, task_id, file_url, file_type, file_name, uploaded_by, uploaded_at.
task_delegation_chain — id, task_id, from_user_id, to_user_id, delegated_at, reason. Tracks every reassignment in order.
SECTION 2: ASSIGNMENT & CHAIN OF COMMAND RULES
Enforce these assignment rules based on role: Admin can assign to anyone. Manager can assign to Crew Lead, Crew Member, New Hire. Crew Lead can assign to Crew Members on their crew only. Crew Member and New Hire cannot assign to others (can create personal tasks for themselves only). HR can assign to Admin and Manager. Sales cannot create tasks. Reassignment: a task can only be passed to someone the current holder has authority over. Original assigner always stays in the notification chain regardless of reassignment. Validate all assignment attempts server-side — return a clear error if a role tries to assign outside their authority.
SECTION 3: TASK CREATION — QUICK ADD AND FULL FORM
Build two creation paths: (1) Quick Add — a persistent floating '+' button accessible from every screen in the app (not just the To-Do module). Tapping it opens a bottom drawer with three fields only: Title (required), Assign To (required, searchable dropdown of eligible users based on role), Priority (required, defaults to Normal). Save button. The entire Quick Add flow must complete in one tap after filling three fields. (2) Full Form — accessible from Quick Add via 'Add more details' link, and from the main To-Do module. Includes all fields described in Section 2. For checklist tasks, show a dynamic list where items can be added, reordered, and deleted. For recurring tasks, show frequency configuration. Both paths save to the same tasks table.
SECTION 4: TASK VIEWS
Build these views in the To-Do module: (1) My Tasks (default) — sections: URGENT & OVERDUE (red, always top), DUE TODAY, DUE THIS WEEK, UPCOMING, NO DEADLINE, COMPLETED (last 30 days, collapsed). Each section shows a count badge. (2) Assigned By Me — tasks the user created, grouped by status. Shows 'awaiting confirmation' items prominently at top. Quick Confirm/Reject buttons. (3) Team View (Manager+ only) — all tasks across the team. Filterable by person, priority, status, category, due date. Toggle between list and Kanban-by-status view. (4) Calendar View — tasks with due dates on the CompanyHQ calendar, color-coded by priority. (5) Task Detail — full task page with all fields, status history timeline, checklist items, comments, attachments, delegation chain. All status changes happen from this page.
SECTION 5: STATUS LIFECYCLE & TRANSITIONS
Implement these status transitions and their rules: assigned → acknowledged (auto when assignee opens task), acknowledged → in_progress (assignee taps 'Start'), in_progress → completed (assignee taps 'Mark Complete', optional note + optional photo upload), completed → confirmed (assigner taps 'Confirm') OR completed → assigned (assigner taps 'Send Back' with required reason note, task re-opens for assignee), any active status → on_hold (requires hold reason), any active status → reassigned (requires new assignee, records delegation chain), any status → cancelled (assigner or admin only, requires reason), system sets status to overdue when current datetime passes due_date+due_time and status is not completed/confirmed/cancelled. Every transition creates a task_history record automatically.
SECTION 6: NOTIFICATION SYSTEM
Build these notification triggers using the existing notification infrastructure (push + in-app + email where specified): On assignment — P1: immediate push to assignee. P2: immediate push. P3/P4: in-app only. Acknowledgment monitoring — cron job every 15 minutes: if P1 task unacknowledged 30+ min, push to assignee AND push to assigner. If P2 unacknowledged 2+ hrs, push to assignee, in-app to assigner at 4hrs. If P3 unacknowledged 24+ hrs, in-app to assignee, notify assigner at 48hrs. Due date reminders — P1: push 1 hour before. P2: push 2 hours before. P3: in-app morning of (7am). All: push 15 min before if specific due time set. Overdue — P1: immediate push to assignee + push to assigner, re-alert every 30 min until resolved. P2: push to assignee, in-app to assigner, re-alert at 2hrs and 24hrs. P3: in-app to assignee, in-app to assigner, re-alert at 24hrs. Any task 3+ days overdue: daily email to manager summarizing all 3+ day overdue tasks with assignee names. Completion: push to assigner when task marked complete (include completion note if present). If requires_confirmation: push to assigner awaiting confirmation. Re-remind at 24hrs if not confirmed. Confirmed: in-app to assignee. Sent back: push to assignee with reason. Daily digest at 7am: push + in-app to each user showing their overdue count, due-today count, unacknowledged count, awaiting-confirmation count.
SECTION 7: RECURRING TASKS
When is_recurring is true, store the recurring config as JSON: { frequency: 'daily'|'weekly'|'monthly'|'custom', days_of_week: [0-6], day_of_month: 1-31, interval_days: number, time: 'HH:MM', advance_days: number }. A cron job runs daily and generates the next task instance when: (a) the previous instance is completed, or (b) the scheduled date arrives regardless of completion status. Missed recurring tasks remain as overdue — they do not silently roll forward. Each generated instance is a new task record linked by parent_task_id. Managers can edit the recurring config, pause a recurring series, or end it.
SECTION 8: ACCOUNTABILITY & REPORTING
Build a Reports tab inside the To-Do module (visible to Admin and Manager only): Individual summary for any user over any date range: total assigned, completed on time (%), completed late (%), overdue/incomplete, average acknowledgment time, average completion time, escalation count. Team overview: completion rate by person (sortable), on-time rate by category, most assigned categories, all currently overdue tasks with days overdue. All reports exportable. Task history is permanent — archived tasks visible in history but removed from active views. Search across all tasks by keyword, assignee, category, date range, status.
SECTION 9: UI REQUIREMENTS
Priority colors must be consistent everywhere: P1 red (#C0392B), P2 orange (#E67E22), P3 yellow (#F1C40F), P4 green (#27AE60). Overdue tasks always display with a red background or red left border — unmistakable. Status indicators: colored dots matching priority, with a clock icon for overdue. The floating Quick Add '+' button must be visible on every screen including mobile. Touch targets for complete/acknowledge buttons must be large enough for use with work gloves — minimum 44px height. Loading states on all async operations. Optimistic UI updates where possible — the tap should feel instant. Works fully on iOS Safari and Android Chrome. Offline capability for My Tasks view — tasks load from cache if offline, sync when connection returns.
SECTION 10: BUILD ORDER
1. Database schema and migrations. 2. Role-based assignment validation. 3. Quick Add flow (floating button, 3-field form). 4. Full task creation form. 5. My Tasks view with all sections. 6. Task detail page with status transitions. 7. Assigned By Me view with confirmation flow. 8. Team view with filters (managers). 9. Notification triggers and cron jobs. 10. Recurring task engine. 11. Calendar integration. 12. Reporting tab. 13. Offline cache for My Tasks. Do not use mock data — all tasks must persist to the real database.
