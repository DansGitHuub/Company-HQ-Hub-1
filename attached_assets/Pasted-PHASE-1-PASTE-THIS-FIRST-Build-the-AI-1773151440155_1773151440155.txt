PHASE 1 — PASTE THIS FIRST
──────────────────────────────────────────────────────────
Build the AI Assistant foundation for CompanyHQ. This is the floating assistant panel, backend chat endpoint, and OpenAI integration. Do not build the full tool library yet — Phase 1 is infrastructure only. Read everything before writing code.
ARCHITECTURE OVERVIEW
The assistant uses OpenAI GPT-4o with function calling. The user sends a message → backend builds a system prompt with the user's identity and role → sends to OpenAI with available tools → OpenAI either responds with text or calls a tool → if a tool is called, the backend executes it and sends the result back to OpenAI → OpenAI gives the final response. The AI is the brain, the app is the hands.
STEP 1: BACKEND — /api/assistant/chat ENDPOINT
Create POST /api/assistant/chat. It receives: { message: string, conversationHistory: array, currentModule: string }. It is authenticated — req.user must be set. Build a three-layer system prompt: Layer 1 — user identity (name, role, permissions summary, current time). Layer 2 — behavioral rules: always confirm before creating/modifying data, never make up records, keep responses short and direct, remember the full conversation, if user asks for something outside their permissions explain what they can't do. Layer 3 — current app state: query the database for this user's overdue tasks count, unread messages count, and any P1 equipment alerts, include these as context. Call OpenAI GPT-4o with: the system prompt, the full conversation history, and function_call: 'auto'. For Phase 1, only include one test tool: navigateTo(module: string) with description 'Navigate the user to a different module in the app'. Handle the response: if OpenAI returns a text response, return it. If OpenAI calls a tool, execute it (for navigateTo, return a navigation instruction), send the result back to OpenAI, return the final response. Always return: { response: string, toolCalled?: string, toolResult?: object, navigationTarget?: string }.
STEP 2: DATABASE — CONVERSATION LOG TABLE
Create table assistant_conversations using Drizzle: id, user_id, session_id (UUID generated per browser session), role ('user' or 'assistant'), content (text), tool_called (nullable), tool_args (JSON, nullable), tool_result (JSON, nullable), tokens_used (nullable), created_at. This table is append-only — never delete entries. Also create table assistant_agents: id, agent_name, agent_key (unique slug), system_prompt_addition, enabled_tools (JSON array of tool names), is_enabled, allowed_roles (JSON array), created_by, updated_at.
STEP 3: FRONTEND — FLOATING ASSISTANT BUTTON AND PANEL
Build a floating assistant component that lives at the root layout level — visible on every page of the app. Collapsed state: a circular button, bottom-right corner, above the Quick Add task button if it exists, with a sparkle/AI icon. Expanded state: slides up from bottom on mobile, right-side panel on desktop. Panel width: 380px on desktop, full width on mobile. The panel does NOT cover the whole screen — the main app remains visible behind it. Inside the panel: a header with 'CompanyHQ Assistant' title, minimize button, and a '...' menu with 'Clear conversation'. A scrollable message thread area. Each user message right-aligned, each AI response left-aligned. A text input at the bottom with a Send button. A microphone icon (visible but inactive and grayed out — labeled 'Voice coming soon' on hover). When the user sends a message: show a typing indicator, call POST /api/assistant/chat, display the response. If the response includes a navigationTarget, navigate the app to that route and show a brief 'Navigating to [module]...' message in the chat.
STEP 4: SPECIAL MESSAGE TYPES
Build these message display components for use in Phase 2: ActionPreviewCard — shows a summary of what the AI is about to do with green Confirm button and red Cancel button. ActionResultCard — shows what was done with a link to the record. SearchResultCard — shows up to 5 search results as tappable rows. ErrorCard — red-tinted card with an error message in plain English. SuggestionChips — a row of tappable quick-reply buttons. These components will be wired up in Phase 2 — just build the display components now.
STEP 5: CONTEXT-AWARE SUGGESTIONS ON OPEN
When the user opens the assistant panel (not on every message), call GET /api/assistant/suggestions. This endpoint queries the database for: overdue tasks count, P1 equipment alerts count, unacknowledged tasks count. Return up to 3 suggestion chips if any of these are non-zero. Examples: 'You have 3 overdue service items — show me', 'You have 2 urgent tasks — summarize them', 'Good morning — what needs attention today?' Display these as tappable SuggestionChips at the top of a new (empty) conversation. Tapping one sends it as a user message automatically.
